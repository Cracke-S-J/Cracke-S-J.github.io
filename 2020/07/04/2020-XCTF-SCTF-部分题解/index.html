指针压缩

v8 8.0引进的；在64位机器中，先申请4G空间的堆用来分配对象，高32位存在r32(x64)里，指针只需32位存低32位。

rce漏洞利用

jit side effects的影响，优化结束后类型混淆->越界读写->aarw->wasm。

因为pointer compression的存在，不能像之前一样无脑通过ArrayBuffer来进行任意读写；但是很容易想到的是可以通过改写数组结构体elements或properties指针的方式实现堆的4GB空间内任意相对地址读写

构造的方式是利用越界读写来找到布置在oobArray后面的BigUint64Array，然后通过越界写覆盖BigUint64Array的base_pointer字段以及external_poiner来实现任意地址读写原语。

PS: 为什么exp中要在最外层包一个main函数：

  • 代码在Reflect.construct函数的内联处理函数ReduceReflectConstruct函数过程中会先将JSCall结点转换成JSConstructWithArrayLike结点；
  • 在ReduceJSConstructWithArrayLike函数中会调用ReduceCallOrConstructWithArrayLikeOrSpread函数；
  • 在ReduceCallOrConstructWithArrayLikeOrSpread函数中如果发现目前优化的函数是最外层函数中的函数的话，则会将结点从JSConstructWithArrayLike转化成JSCallForwardVarargs结点，从而最终不会出现JSCreate结点；
  • 漏洞点是JSCreate结点存在的side-effect（回调函数）造成的。

改context开启mojo

但是这里不需要rce，由于有sandbox，所以只需要打开mojo，让mojo去execute code。

Looking through the Chrome source code, we can see that the Mojo bindings are added to the Javascript context in RenderFrameImpl::DidCreateScriptContext, based on the member variable enabled_bindings_. So, to mimic the command line flag we can use our read/write to set that value to BINDINGS_POLICY_MOJO_WEB_UI, and force the creation of a new ScriptContext for the main frame and we should have access to the bindings!
It’s slightly painful to get hold of the RenderFrameImpl for the current frame, but by following a chain of pointers from the global context object we can locate chrome_child.dll, and find the global g_frame_map, which is a map from blink::Frame pointers to RenderFrameImpl pointers. For the purposes of this exploit, we assume that there is only a single entry in this map; but it would be simple to extend this to find the right one. It’s then trivial to set the correct flag and reload the page - see enable_mojo.js for the implementation.

确定chrome_child.dll的地址->手动查找偏移->得到RenderFrameImpl指针->aaw修改context的flag。

chrome_child.dll基址确定方法：leak出一个blink对象div的地址，它偏移0x20的位置是HTMLDivElement对象，读出后，再读出它首部的虚表地址，然后减去和chrome_child.dll的偏移就是chrome_child.dll的基地址了。

Mojo部分

漏洞有两个：

  1. UAF，很明显FreeBuffer调用的时候没清空指针。
  2. OOB read，首先通过put吧size_设大，大于0x1000，然后get就能OOB读，这样可能可以读到例如堆地址之类的东西，这个具体得调试。

思路大概是先leak堆地址，UAF之后通过get的new std::string(buffer_, sizeof(MojoBuffer))占坑，然后此时把虚表写成堆上构造的vtable就能控rip，控了rip就简单了，pivot之后ROP到WinExec或者system读flag。这里注意windows的DLL地址都是一样的，所以可以直接从renderer泄露各种DLL的地址。

还有一点注意就是get和put都是虚函数，直接在free的object上调用他们会崩，所以得弄两个instance来弄。

leak堆地址的方案我想到个可能比较可行的，就是通过这个new std::string来spray，然后因为size小的std::string的结构是指针+data，我们就可以通过搜data来泄露指针。
